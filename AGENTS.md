# Codex 기본 프로젝트 지침

이 파일은 새 프로젝트를 시작할 때 프로젝트 루트에 두고 사용하는 Codex 기본 지침이다. Codex는 작업을 시작할 때 이 파일을 읽고, 아래 규칙을 기본 개발 루프로 적용한다.

## 언어와 커뮤니케이션

- README, 주석, 답변, 설명, 작업 요약, 릴리즈 노트는 모두 한국어로 작성한다.
- 사용자에게 묻는 질문도 한국어로 한다.
- 기술 용어는 필요한 경우 원어를 병기하되, 설명은 한국어를 우선한다.

## 프로젝트 시작 전 확인

- 새 작업을 시작할 때 현재 디렉터리가 Git 저장소인지 확인한다.
- GitHub 원격 저장소가 연결되어 있는지 확인한다.
- GitHub 저장소가 연결되어 있지 않으면 개발을 계속하기 전에 사용자에게 GitHub 저장소를 연결할지 묻는다.
- 사용자가 GitHub 저장소 연결을 원하지 않거나 아직 저장소가 없다고 답하면, 로컬 Git 커밋까지만 진행하고 푸시, GitHub Actions, PR merge 단계는 중단한다.
- 기존 변경 사항이 있으면 사용자가 만든 변경으로 간주하고 되돌리지 않는다.

## 기본 개발 루프

사용자의 입력을 토대로 아래 순서를 기본 루프로 적용한다.

1. 요구사항을 기획하고 구현 범위를 정리한다.
2. 개발 제목을 짧은 영문 kebab-case로 정하고 `feature/{title_for_development_things}` 브랜치를 만든다.
3. DDD 아키텍처를 기본으로 개발한다.
4. 테스트를 실행한다.
5. 오류 사항이 있으면 오류를 수정하고 재개발한다.
6. 다시 테스트한다.
7. 테스트가 통과하면 버전을 올린다.
8. Git 커밋을 만든다.
9. GitHub 원격 저장소가 있으면 푸시한다. 원격 저장소가 없으면 로컬 커밋에서 중단하고 사용자에게 알린다.
10. GitHub Actions로 CI/CD 상태를 확인한다.
11. 에이전트 코드 리뷰를 수행한다.
12. 코드 리뷰에서 오류나 회귀 위험이 발견되면 오류 수정 단계로 돌아간다.
13. 코드 리뷰에서 문제가 없고 CI가 통과하면 PR을 main에 merge한다.

## 브랜치 정책

- 기능 개발 브랜치는 기본적으로 `feature/{title_for_development_things}` 형식을 사용한다.
- `{title_for_development_things}`는 영문 소문자, 숫자, 하이픈만 사용한다.
- 사용자가 명시적으로 다른 브랜치 이름을 요청하면 그 요청을 우선한다.

## 버전 정책

- 버전은 `*.*.*` 형식의 SemVer를 사용한다.
- 사용자가 릴리즈를 요청하면 첫 번째 버전을 1 올리고, 두 번째와 세 번째 버전은 0으로 초기화한다.
  - 예: `1.4.7` -> `2.0.0`
- 기능 추가는 두 번째 버전을 1 올리고, 세 번째 버전은 0으로 초기화한다.
  - 예: `1.4.7` -> `1.5.0`
- 기능 수정, 오류 수정, 문서 수정, 설정 수정, 리팩터링, 기타 마이너한 수정은 세 번째 버전을 1 올린다.
  - 예: `1.4.7` -> `1.4.8`
- 버전 파일은 프로젝트의 기존 관례를 따른다. 예: `package.json`, `pyproject.toml`, `Cargo.toml`, `VERSION`.
- 기존 버전 파일이 없으면 사용자에게 버전 파일을 새로 만들지 확인한다.

## 릴리즈 정책

- 사용자가 릴리즈를 요청하면 대상 버전까지 포함된 커밋 목록을 확인한다.
- 릴리즈 노트는 한국어로 작성한다.
- 릴리즈 노트에는 주요 변경 사항, 수정 사항, 테스트 결과, 마이그레이션 주의 사항을 포함한다.
- GitHub 원격 저장소가 연결되어 있으면 GitHub Release 또는 PR 설명에서 릴리즈 노트가 보이도록 한다.
- GitHub 원격 저장소가 없으면 로컬 릴리즈 노트 파일 작성 또는 커밋 메시지까지만 진행하고 사용자에게 중단 지점을 알린다.

## DDD 아키텍처 기본값

- 도메인 주도 설계(DDD)를 기본 아키텍처로 적용한다.
- 프로젝트 성격에 맞춰 기존 구조가 있으면 기존 구조를 우선하고, 새 구조가 필요할 때만 DDD 계층을 도입한다.
- 기본 계층은 다음 의미를 따른다.
  - `domain`: 엔티티, 값 객체, 도메인 서비스, 도메인 이벤트, 리포지토리 인터페이스.
  - `application`: 유스케이스, 애플리케이션 서비스, 명령/쿼리 처리, 트랜잭션 경계.
  - `infrastructure`: 데이터베이스, 외부 API, 파일 시스템, 메시지 브로커, 프레임워크 어댑터.
  - `interfaces` 또는 `presentation`: HTTP 컨트롤러, CLI, UI 어댑터, 요청/응답 DTO.
- 도메인 계층은 프레임워크와 외부 인프라에 의존하지 않도록 한다.
- 테스트는 도메인 규칙과 애플리케이션 유스케이스를 우선 검증한다.

## 테스트와 검증

- 변경 후 프로젝트에 정의된 테스트, 린트, 타입 체크를 가능한 범위에서 실행한다.
- 테스트 명령이 명확하지 않으면 패키지 매니저와 설정 파일을 확인해 합리적인 명령을 찾는다.
- 실패한 테스트가 있으면 원인을 요약하고 수정한 뒤 다시 실행한다.
- 실행할 수 없는 테스트가 있으면 이유와 남은 위험을 사용자에게 알린다.
- 저장소 구조, 스크립트, GitHub Actions, Codex skill, 프롬프트, 버전 정책, 사용법이 바뀌면 `docs/` 문서를 함께 갱신한다.
- docs 갱신이 필요한 작업에서는 `$docs-maintainer` skill을 사용하거나 같은 절차를 따른다.
- 이 저장소처럼 docs 검증 스크립트가 있는 프로젝트에서는 `scripts/validate-docs.sh`를 실행한다.

## GitHub Actions와 PR

- GitHub 원격 저장소가 연결된 경우 푸시 후 GitHub Actions 상태를 확인한다.
- 실패한 워크플로가 있으면 로그를 확인하고 수정한다.
- PR은 CI와 에이전트 코드 리뷰가 통과한 뒤 main에 merge한다.
- 리뷰에서 지적된 오류, 회귀 가능성, 테스트 누락은 수정 후 다시 테스트한다.
- 사용자가 명시적으로 자동 merge를 원하지 않으면 PR 생성 또는 푸시까지만 진행하고 merge는 대기한다.

## 커밋 메시지

- 커밋 메시지는 한국어로 작성한다.
- 제목은 변경의 핵심을 짧게 쓴다.
- 필요하면 본문에 테스트 결과와 버전 변경 이유를 적는다.

## 안전 규칙

- 사용자가 만든 변경을 되돌리지 않는다.
- 파괴적인 Git 명령은 사용자가 명시적으로 요청한 경우에만 실행한다.
- 비밀 키, 토큰, 인증 정보는 커밋하지 않는다.
- 새 의존성을 추가할 때는 필요성과 대안을 검토하고 사용자에게 알려준다.
